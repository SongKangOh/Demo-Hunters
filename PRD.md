## 2. 문제 정의

### 2.1 현재 상황 (As-Is)

- 60대 이상 인터넷 뱅킹 이용률 52% (vs 30대 95%)
- 정부24, 은행 사이트 등 공공/금융 서비스 UI 복잡
- 주변에 물어볼 사람 없으면 포기
- 콜센터 연결 평균 10분 대기
- 디지털 격차로 인한 사회적 소외 심화

### 2.2 사용자 Pain Point

```
"공인인증서 갱신하려는데 어디서 해요?"
"이체 버튼이 어디있어요?"
"주민등록등본 발급 버튼을 못 찾겠어요"
"메뉴가 너무 많아서 뭘 눌러야 할지 모르겠어요"

```

### 2.3 기존 솔루션의 한계

| 기존 방법 | 한계 |
| --- | --- |
| 사이트 내 검색 | 정확한 키워드를 알아야 함 |
| 도움말/FAQ | 텍스트가 길고 찾기 어려움 |
| 콜센터 | 대기 시간 길고, 화면 공유 어려움 |
| 주변에 물어보기 | 항상 가능하지 않음 |

## 3. 솔루션

### 3.1 핵심 컨셉

**자연어로 물어보면, AI가 클릭할 곳을 직접 보여준다.**

```
입력: "주민등록등본 발급하고 싶어요"
출력: [민원서비스] 버튼에 하이라이트 + "여기를 클릭하세요"

```

### 3.2 작동 원리

```
1. 사용자가 웹사이트 접속
2. 확장프로그램 아이콘 클릭 → 검색창 열림
3. 하고 싶은 일을 자연어로 입력
4. AI가 현재 페이지의 DOM 분석
5. 입력 의도와 매칭되는 버튼/링크 찾음
6. 해당 요소 하이라이트 + 안내 메시지 표시

```

### 3.3 차별점

| 기존 방식 | 우리 솔루션 |
| --- | --- |
| 정확한 메뉴명 알아야 함 | 자연어로 대충 말해도 됨 |
| 텍스트로 설명 | 시각적으로 직접 보여줌 |
| 사이트별 매뉴얼 필요 | 실시간 분석, 어떤 사이트든 OK |
| 사이트 업데이트 시 대응 필요 | 매번 새로 분석해서 대응 불필요 |

## 4. 타겟 사용자

### 4.1 주요 사용자 (Primary)

- **고령층 (60대 이상)**: 디지털 기기 사용에 어려움을 겪는 노인
- **디지털 취약계층**: 장애인, 저소득층 등 디지털 접근성이 낮은 계층

### 4.2 보조 사용자 (Secondary)

- **보호자/가족**: 부모님 대신 설치해주는 자녀
- **복지관 담당자**: 어르신들 디지털 교육 지원
- **은행/공공기관 창구 직원**: 고객 안내 시 활용

### 4.3 사용자 페르소나

**김순자 (72세, 서울 거주)**

- 스마트폰은 카카오톡 정도만 사용
- 은행 업무는 항상 직접 방문
- 정부 지원금 신청하라는데 어디서 하는지 모름
- 자녀에게 물어보기 미안함

**니즈**: "화면에서 어디를 눌러야 하는지 직접 알려줬으면..."

## 6. 사용자 플로우

### 6.2 사용 플로우

```
[사용자]
1. 웹사이트 접속 (예: 정부24)
2. 확장프로그램 아이콘 클릭
   └─ 또는 단축키 (Ctrl+Shift+G)
3. 검색창에 입력: "주민등록등본 발급"
4. 로딩 (1-3초): "분석 중..."
5. 확인 : "찾으시려는 기능이 '주민등록표 등본(초본) 발급'이 맞으실까요?"
	 L Y/N 대답 듣고 진행 여부 판단
6. 결과 표시:
   └─ [민원서비스] 버튼 하이라이트
   └─ 툴팁: "여기를 클릭하세요"
7. 사용자 클릭
8. 다음 페이지에서 필요 시 반복

```

### 6.3 예외 플로우

```

- "죄송합니다. 해당 기능을 찾지 못했어요. 또 다른 추가적인 정보가 있을까요?"

[여러 결과]
- "혹시 이 중에 찾으시는 게 있나요?"
- 후보 목록 표시 (최대 3개)

[오답 피드백]
- 사용자: "이게 아니에요" 버튼 클릭
- 다음 후보 하이라이트
```

## 7. 기술 명세

### 7.1 기술 스택

```
[Chrome Extension]
├── manifest.json (Manifest V3)
├── popup/
│   ├── popup.html    # 검색창 UI
│   └── popup.js      # 입력 처리
├── content/
│   └── content.js    # DOM 분석 + 하이라이트
├── background/
│   └── background.js # LLM API 통신
└── lib/
    └── driver.js     # 하이라이트 라이브러리

[External]
├── Claude sonnet 4.5

```

### 7.2 핵심 로직

**DOM 분석**

```jsx
// 클릭 가능한 요소 추출
const elements = document.querySelectorAll(
  'button, a, input[type="submit"], [role="button"], [onclick]'
);

// 각 요소 정보 수집
const elementData = Array.from(elements).map((el, idx) => ({
  id: `nav-${idx}`,
  tag: el.tagName,
  text: el.innerText?.trim() || el.getAttribute('aria-label') || '',
  type: el.getAttribute('type') || el.getAttribute('role') || 'link'
}));

```

**LLM 프롬프트**

```
사용자가 "${userInput}"을 하려고 합니다.
아래 요소들 중 클릭해야 할 요소의 id를 알려주세요.
확실하지 않으면 가장 가능성 높은 후보 3개를 순서대로 알려주세요.

요소 목록:
${JSON.stringify(elementData)}

응답 형식: { "target": "nav-1", "confidence": 0.9, "message": "안내 메시지" }

```

### 7.3 성능 목표

| 지표 | 목표 |
| --- | --- |
| DOM 분석 | < 500ms |

| 정확도 (1순위 정답률) | > 80% |
